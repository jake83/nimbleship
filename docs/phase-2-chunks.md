# Phase 2 parallel chunks

Executable specs for the Phase 2 fan-out. Each chunk is one PR by one agent
in an isolated worktree. Read CLAUDE.md (the development loop applies -
TDD, pipeline, human merge), CONTEXT.md (use its terms exactly), and ADRs
0003/0007/0008 before starting. The foundations these build on: the
evaluator registry (`api/src/nimbleship/domain/checks/`), the Phase 2
fields already present on `ServiceDeclaration`/`Shipment`
(`domain/model.py`), and the band stubs (`domain/costs.py`,
`domain/charges.py`).

Shared rules for every chunk:
- New checks follow `checks/dimension.py`: unknown shipment fact = ok=True
  with actual "unknown (optimistic)" (ADR 0007); one module per kind; add to
  ALL_CHECKS; update the trace test's expected check-name set.
- Rulebook data changes need no Alembic migration (JSON column); real table
  additions need an autogenerated migration plus the honesty test passing.
- Do not touch files another chunk owns; shared-file edits (model.py,
  ALL_CHECKS registration, trace test) must stay additive and minimal.

## A. Geography

Goal: shipping areas as data; area facts on the shipment; area checks.
- Tables (+ migration): `shipping_areas` (code, name, country), 
  `postcode_areas` (postcode prefix -> area code). CRUD API under
  `/api/shipping-areas` (list/create/update), validated, no auth (matches
  existing surfaces).
- A resolver: destination postcode + country -> list of area codes.
  EVERY matching prefix counts (old-system parity; a specific prefix never
  shadows a general one). The longest-prefix trick is only the query
  optimisation ported from `getBlockedHauliersForPostcode`: one query with
  all prefixes.
- `create_consignment` populates `Shipment.shipping_areas` via the resolver.
- New checks: `area_blocked` (shipment area in service.areas_blocked ->
  block), `area_served` (service.areas_served not None and no overlap ->
  block). Empty shipping_areas = no areas matched = optimistic.
- Out of scope: surcharges (dead table in the old system), UI.

## B. Delivery Propositions

Goal: the customer promise as a first-class filter (CONTEXT.md term).
- Table (+ migration): `delivery_propositions` (code, name, description) +
  CRUD API under `/api/propositions`. Seed a demo set (next-day, economy).
- New check: `proposition` - shipment.proposition None = optimistic;
  service.propositions [] = unrestricted; otherwise membership.
- `ConsignmentIn` gains optional `proposition`; flows into the Shipment.
- Validation: a draft rulebook naming a proposition code that does not
  exist in the catalogue is rejected at draft time (fail at authoring, not
  at allocation).
- Out of scope: checkout endpoint (integration step), legacy serviceGroup
  mapping (Phase 4 edge).

## C. Delivery Costs

Goal: banded costs replace the flat number in the selection policy.
- Harden `domain/costs.py`: validators (band type implies its fields),
  then a pure `calculate_cost(bands, shipment) -> Decimal | None` porting
  the old semantics: weight bands (base + additional per kg over min),
  parcel bands, fuel surcharge percentage over the subtotal, dimension
  surcharge. None = no applicable band (service excluded from selection,
  flagged in the trace as "no cost data" - ADR 0007: missing cost data is
  loud, never silent).
- Selection policy: cheapest calculated cost; services with cost_bands use
  the calculator, services with only flat `cost` keep using it (migration
  path); tie-break unchanged.
- New check to add while here (dimension-adjacent): `girth` following
  checks/dimension.py against `max_girth_cm`.
- Out of scope: charges (chunk D), currency (GBP assumed, as old system).

## D. Delivery Charges

Goal: what the customer pays, computed per service (CONTEXT.md: Delivery
Charge, never confused with Cost).
- Harden `domain/charges.py`: validators, then pure
  `calculate_charge(bands, shipment, areas) -> Decimal | None` porting old
  semantics: scope precedence area -> country -> all (first scope with a
  matching band wins), base + additional per kg over band minimum.
- An API projection: given a shipment-shaped payload, return eligible
  services with their charges (the seed of the checkout endpoint; mount at
  `/api/quotes` - the integration step will shape the final contract).
- Out of scope: warehouse scoping (needs chunk E; leave the field unused),
  lead times/delivery dates.

## E. Warehouses

Goal: the logical dispatch identity (CONTEXT.md: Warehouse).
- Tables (+ migration): `warehouses` (code, name, sender address fields),
  `warehouse_collection_days` (weekday flags per carrier or global - check
  the old system's shape and port faithfully), `warehouse_holidays` (date,
  description). CRUD API under `/api/warehouses`.
- `ConsignmentIn` gains optional `warehouse` code; stored on the
  consignment; label sender details come from the warehouse when present
  (Drop Out label currently hardcodes nothing - it uses request fields, so
  thread warehouse sender address through `LabelRequest`).
- Dispatch-date helper: next collection day from a warehouse calendar
  (skip holidays + non-collection days) - pure function, well tested.
- Out of scope: per-website mapping (needs the sales-platform work),
  cutoff times (port later with delivery dates).

## F. Rules UI

Goal: the versions/draft/dry-run/publish workflow, visible.
- Web app (React/shadcn, existing conventions): a Rulebook section with:
  versions list (version, status, author, created), version detail with a
  diff against the previous version (client-side JSON diff is fine),
  draft editor (form-based service editing - no free-text JSON), dry-run
  runner (order numbers or recent-N; results table with changed rows
  highlighted), publish button with confirmation showing the dry-run
  summary.
- API additions only if the UI genuinely needs them (e.g. GET single
  version content - add it if missing).
- Component tests for the editor validation states and the dry-run table;
  keep the API mocked in tests.
- Out of scope: auth, AI drafting (Phase 5), design-system work beyond
  existing shadcn patterns.
